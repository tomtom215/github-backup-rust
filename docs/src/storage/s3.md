# S3-Compatible Storage

`github-backup` can sync JSON metadata (and optionally release assets) to any
S3-compatible object store after the primary local backup completes.

## Supported Providers

| Provider | Notes |
|---------|-------|
| **AWS S3** | Native; set `--s3-region` |
| **Backblaze B2** | Use S3-compatible endpoint |
| **Cloudflare R2** | Zero egress fees; use custom endpoint |
| **MinIO** | Self-hosted; use custom endpoint |
| **DigitalOcean Spaces** | Use `nyc3.digitaloceanspaces.com` endpoint |
| **Wasabi** | Use `s3.wasabisys.com` endpoint |

## Basic Usage

```bash
github-backup octocat \
  --token "$GITHUB_TOKEN" \
  --output /backup \
  --all \
  --s3-bucket my-backup-bucket \
  --s3-region us-east-1 \
  --s3-access-key "$AWS_ACCESS_KEY_ID" \
  --s3-secret-key "$AWS_SECRET_ACCESS_KEY"
```

Or via environment variables:

```bash
export AWS_ACCESS_KEY_ID=AKIAIOSFODNN7EXAMPLE
export AWS_SECRET_ACCESS_KEY=wJalrXUtnFEMI/K7MDENG/bPxRfiCYEXAMPLEKEY

github-backup octocat --token "$GITHUB_TOKEN" --output /backup --all \
  --s3-bucket my-backup-bucket
```

## Provider-Specific Examples

### Backblaze B2

```bash
github-backup octocat --token $GITHUB_TOKEN --output /backup --all \
  --s3-bucket my-b2-bucket \
  --s3-region us-west-004 \
  --s3-endpoint "https://s3.us-west-004.backblazeb2.com" \
  --s3-access-key "$B2_KEY_ID" \
  --s3-secret-key "$B2_APP_KEY"
```

### Cloudflare R2

```bash
github-backup octocat --token $GITHUB_TOKEN --output /backup --all \
  --s3-bucket my-r2-bucket \
  --s3-region auto \
  --s3-endpoint "https://<ACCOUNT_ID>.r2.cloudflarestorage.com" \
  --s3-access-key "$R2_ACCESS_KEY" \
  --s3-secret-key "$R2_SECRET_KEY"
```

### MinIO (self-hosted)

```bash
github-backup octocat --token $GITHUB_TOKEN --output /backup --all \
  --s3-bucket github-backup \
  --s3-region us-east-1 \
  --s3-endpoint "http://minio.internal:9000" \
  --s3-access-key minioadmin \
  --s3-secret-key minioadmin
```

## Key Prefix

Organise objects under a prefix within the bucket:

```bash
--s3-prefix "backups/github/"
```

All objects will be stored as `backups/github/<owner>/json/...`.

## Including Release Assets

By default, only JSON metadata is synced to S3.  Add `--s3-include-assets` to also upload binary release assets:

```bash
--s3-bucket my-bucket --s3-include-assets
```

> **Warning**: Binary assets can be large.  Use a lifecycle rule in S3 to manage costs.

## Incremental Sync

`github-backup` uses `HeadObject` to check each file before uploading.  Files are
skipped if they already exist in S3 at the same key.  This makes subsequent runs
fast: only changed JSON files and new release assets are uploaded.

## S3 Flags Reference

| Flag | Default | Description |
|------|---------|-------------|
| `--s3-bucket` | *(required)* | Bucket name |
| `--s3-region` | `us-east-1` | AWS region |
| `--s3-prefix` | *(empty)* | Object key prefix |
| `--s3-endpoint` | *(AWS default)* | Custom S3-compatible endpoint URL |
| `--s3-access-key` | `AWS_ACCESS_KEY_ID` env | Access key ID |
| `--s3-secret-key` | `AWS_SECRET_ACCESS_KEY` env | Secret access key |
| `--s3-include-assets` | `false` | Upload binary release assets |

## Implementation Details

The S3 client uses a **pure-Rust SigV4 implementation** (no AWS SDK, no openssl).
Authentication is provided by HMAC-SHA256 signatures computed with the `sha2` and
`hmac` crates.  This keeps the binary small and avoids OpenSSL linkage issues.
